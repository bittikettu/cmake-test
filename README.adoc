== Introduction

VFD-9000 is a single-player escape room played inside a fake unix shell that
is rendered to look like an old vacuum fluorescent display — phosphor glow,
curved glass, scanlines and flicker, all driven by a CRT post-processing
shader. You wake up locked in the cold storage of *Koivu & Sons*; the only
way out is the door keypad and a dusty terminal. Explore the virtual
filesystem, decrypt the clue files, bring the locked-down systems back online,
and let yourself out.

Written in C99 with https://www.raylib.com/[raylib]. Escape rooms ("DLC") are
authored as runtime-loaded Lua files, so new rooms need no engine changes. The
project also doubles as a modular CMake scaffold (git-derived versioning, a
`BUILD_PHASE` library/app switch).

== Lore

The *VFD-9000* was never meant to be the last thing you'd see. It was a cheap
door-control terminal — one board, an 8 MHz CPU, a vacuum-fluorescent panel
bright enough to read through freezer fog — bolted next to ten thousand doors
across as many businesses. *Doorctl* watched the keypad, *autolock* slammed the
bolt the moment a door swung shut, and a quiet line in every brochure promised
that the firmware would "never fail safe in a way that compromises the
premises." It failed safe by sealing you *in*.

So the machines outlived their makers, and the doors kept locking. The cold
stores of *Koivu & Sons* froze a careless guest out of daylight. *Helios
Datacenter* dropped a high-voltage maglock across Cage 4 with a contractor
still inside. *Vertex Softworks* wired its dev-floor exit to the build pipeline
and forgot anyone might be working late. Each site rigged its own release ritual
on top of the same stubborn terminal — a kernel module to coax, a daemon to
revive, a database to interrogate, a code review to land — and each one left,
somewhere on the disk, a half-finished note from the person who got out before
you. Or didn't.

There is always a way through the shell. The door always opens for the right
four digits. The VFD-9000 keeps its promise — eventually.

== Requirements

* CMake (>= 3.20) and a build generator (Ninja).
* A C compiler — the presets use Clang/LLVM.
* `git` and `git-lfs` (the audio is stored via Git LFS, see below).
* raylib *and Lua* are fetched and built automatically through CMake
  `FetchContent`; no manual install is required.
* For the browser build only: Docker (the Emscripten toolchain ships in the
  image — no local emcc needed).

On Linux, install the X11/Wayland/GL development packages raylib needs, e.g.
on Debian/Ubuntu:

[source,sh]
----
sudo apt install build-essential cmake ninja-build git git-lfs pkg-config \
    libx11-dev libxrandr-dev libxinerama-dev libxcursor-dev libxi-dev \
    libgl1-mesa-dev libglu1-mesa-dev libwayland-dev libxkbcommon-dev \
    wayland-protocols
----

== Git LFS

The audio files (`src_raylib/*.mp3` — boot chime, room hum, door foley) are
tracked with https://git-lfs.com/[Git LFS] (see `.gitattributes`). Install LFS
once before cloning so the real audio is pulled instead of pointer files:

[source,sh]
----
git lfs install
git clone git@github.com:bittikettu/cmake-test.git
----

If you cloned before installing LFS, run `git lfs pull` to fetch the audio.

== Build and run

The build presets live in `CMakePresets.json` (Ninja + Clang).

[source,sh]
----
# Release build (what you normally run)
cmake --preset local_application_release
cmake --build build/local_application_release
build/local_application_release/src_raylib/modular_cmake-raylib-app.exe

# Debug build, one shot (configure + build)
cmake --workflow --preset app
----

raylib and Lua are compiled on the first configure (slow once, cached after).
The terminal font is embedded in the binary; the audio (`*.mp3`,
`separated_keypresses/`) and the DLC rooms (`src_raylib/rooms/*.lua`) are
staged next to the executable at build time.

=== Web (WASM) build

The browser build is produced with Emscripten through Docker (see `Dockerfile`):

[source,sh]
----
# Pass the host's git version so the on-screen firmware version is correct
# (the container's git can't read the COPYed tree's tags on its own).
docker build --build-arg GIT_VERSION="$(git describe --tags --always --dirty)" \
    -t vfd-9000-web .                     # full build, served by nginx
docker run --rm -p 8080:80 vfd-9000-web   # -> http://localhost:8080
----

It emits `index.{html,js,wasm,data}`; the rooms and audio are packed into the
`.data` preload bundle and read from the in-memory filesystem at runtime.

NOTE: The bezel firmware version is derived from `git describe` at configure
time. In the browser build it is passed in via the `GIT_VERSION` build arg
above; omit the arg and the container falls back to its own git (it marks the
copied tree as a safe repository so it can still read the tags).

== How to play

1. Boot: an 8-second *Koivu & Sons* splash plays over the boot chime, then the
   terminal runs its self-test.
2. Pick a *service cartridge* (escape room) from the menu.
3. Log in with the account that sets your difficulty:
+
[cols="1,4"]
|===
| `n00b` | guided — files tell you which command to use next
| `l33t` | no hints, no `help` — relies on your own unix knowledge
|===
4. Use the shell to recover the 4-digit door code and satisfy the room's gate,
   then `unlock <code>`. Type `help` (or `ls /usr/bin`) to see the installed
   tools: `ls cat grep mv tar base64 rot13 modprobe lsmod dmesg service sql
   unlock`, and more.

[cols="1,3"]
|===
| typing + Enter | shell / menu input
| Up / Down      | command history
| Mouse wheel, PgUp / PgDn | scrollback
| F11 or Alt+Enter | fullscreen toggle
| Esc            | power off (quits)
|===

== Architecture

The game lives in `src_raylib/` and is split along a *host vs. game* seam:
*C is the host*, and *all game logic is Lua*. New rooms — and new verbs — need
no recompile.

* `main.c` — the host: terminal scrollback + teletype reveal, the CRT/VFD
  rendering and shader, the on-screen keyboard, audio, input editing, and the
  SPLASH/BOOT animation. After boot it drives the Lua game: each input line goes
  to `game_submit`, and the prompt + sub-mode come from `game_prompt`/`game_mode`.
* `engine.h` — the small host API main.c exposes (terminal output, audio,
  `boot_start`, the `b64encode`/`rot13` string helpers).
* `lua_rooms.c` — the Lua host bridge: a sandboxed VM exposing `vfd.*` (C string
  helpers) and `host.*` (print/clear/play/reboot/time/rand), which loads
  `game.lua` and every `rooms/*.lua` and bridges the loop.
* `game.lua` — the whole game in Lua: the virtual filesystem, every shell verb,
  the `select -> login -> shell -> win` state machine, and the gate/win logic.
  It carries a scripted self-test that plays a full win at startup.
* `rooms/*.lua` — the content packs ("DLC"), plain Lua tables read by `game.lua`.
  `coldstore.lua` (Cold Storage) is the template; `coldvault.lua` (Cold Vault,
  a database room) is a second worked example.

To add a room: copy `src_raylib/rooms/coldstore.lua`, rewrite its filesystem
table and text, and give it a unique `id`/`title`. On native you can drop the
file into the `rooms/` folder next to the executable with no rebuild; it
appears in the boot menu automatically. The puzzle gate is data — `winFlags`
plus an optional `modprobe` module gate, `service` daemon gate, file `mv`, or
`sql` database — so most rooms need no engine changes. See `CLAUDE.md` for
detail.
